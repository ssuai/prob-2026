# **Problem 2(Intermediate):**   **Kubernetes Memory Bounding (Markov Inequality)**

##### **\[Condition]**

##### A global e-commerce platform utilizes Kubernetes to orchestrate a distributed microservice responsible for cart checkout processing. Due to highly complex underlying database dependencies, the exact probability density function (PDF) of the memory consumed by a single pod is unknown to the engineering team. However, the internal Kubernetes metrics server confirms that the pod's average memory consumption over a 30-day period is exactly $\\mu = 250$ Megabytes (MB). The Kubernetes cluster scheduler must establish a hard memory limit for the pod's cgroup. If the pod exceeds this strict threshold, the Linux OOM-killer will instantly terminate it, resulting in failed customer transactions and revenue loss.

##### **\[Problem]**

1. ##### State the fundamental mathematical condition required to apply Markov's Inequality to this specific orchestration metric.
2. ##### The infrastructure team proposes setting a hard memory limit of $1000$ MB to conserve physical node capacity across the cluster. Using Markov's Inequality, calculate the mathematical upper bound on the probability that a pod will exceed this limit and be terminated.
3. ##### Following the previous scenario, the DevOps team has determined that a 25% probability of termination is unacceptable as it violates the Service Level Agreement (SLA) for the checkout process. To maximize system stability, the team aims to keep the mathematical upper bound of the probability that a pod is terminated by the OOM-killer below 0.1% ($0.001$). The average memory consumption of the pod remains constant at $\\mu = 250$ MB. Using Markov's Inequality, calculate the minimum hard memory limit ($a$) that must be established to satisfy this strict reliability requirement. State your final answer in Gigabytes (GB). (Assume $1 \\text{ GB} = 1,000 \\text{ MB}$ for simplicity.)
